## Critical Problems (Критические проблемы)

### C-01: Частичная интеграция state.py
**Описание:** В `handlers.py` и `main.py` смешано использование глобальных переменных (`current_mode`, `student_level`) и передачи через параметры. `state.py` создан, но не используется последовательно.

**Последствия:** Сложность поддержки, потенциальные баги при передаче состояний.

**Решение:** Полностью перевести на `get_state()`/`set_state()`, убрать передачу mode/level как аргументы функций.

**Файлы:** `main.py`, `handlers.py`, `state.py`

**Статус:** Not started

### C-02: Циклические импорты
**Описание:** 
- `main.py` импортирует `handlers.py`
- `handlers.py` импортирует `get_learning_context` из `main.py`
- `handlers.py` импортирует `code_review_function`, которая импортирует `LLM` из `config.py`
- `config.py` теперь импортирует `dotenv`, что нормально, но при импорте в других местах может быть порядок проблем

**Последствия:** Сложность изменений, потенциальные ImportError.

**Решение:** Вынести общие утилиты (`get_learning_context`, `get_news_context`) в отдельный модуль `common.py` или `context.py`. Убедиться, что импорты идут только в одном направлении.

**Файлы:** `main.py`, `handlers.py`, `code_review.py`, `config.py`

**Статус:** In progress (частично исправлено: убран импорт `get_llm` из handlers в пользу передачи как параметра)

### C-03: Пустые/заглушечные файлы
**Описание:** 
- `db_operations.py` — полностью пустой (только pass). Функциональность дублируется в `memory.py`.
- `generators.py` — вызывает неопределённые функции (`extract_json_block`, `get_relevant_docs`, `print_panel`), отсутствуют импорты.

**Последствия:** Запуск generators.py приведёт к NameError. Наличие db_operations.py сбивает с толку.

**Решение:** 
- Удалить `db_operations.py` (выполнено)
- Исправить `generators.py` (добавить импорты, реализовать недостающие функции) — **выполнено**

**Файлы:** `db_operations.py` (удалён), `generators.py` (исправлен)

**Статус:** Done (generators.py), Done (db_operations.py удалён)

### C-04: Отсутствие проверки Docker
**Описание:** В `practice.py` функции `exec_in_container`, `run_docker_cmd` не проверяют, доступен ли Docker демон. Если Docker не запущен, программа упадёт с исключением.

**Последствия:** Падение при попытке использовать лабы без запущенного Docker.

**Решение:** Добавить `docker_available()` в начале, выводить понятное сообщение "Docker недоступен. Запустите Docker Desktop."

**Файлы:** `practice.py`

**Статус:** Not started

### C-05: Валидация команд Docker exec
**Описание:** В `practice.py:exec_in_container` команда пользователя (`cmd`) передаётся напрямую в `docker exec`. Риск command injection, если команда формируется неправильно или пользователь может ввести критический ввод.

**Последствия:** Потенциальный escape из контейнера, выполнение произвольных команд на хосте.

**Решение:** 
- Использовать `shlex.split()` и передавать команду как список аргументов, а не строку.
- Ограничить команды белым списком разрешённых (например, только базовые: ls, cat, grep, find, ps и т.д.).
- Или использовать `docker exec` с аргументами, а не через shell=True.

**Файлы:** `practice.py:exec_in_container`

**Статус:** Not started

## Architecture Issues (Архитектурные недостатки)

### A-01: Неиспользование state.py в handlers
**Описание:** `handlers.py` принимает `mode` и `student_level` как параметры, хотя `state.py` уже реализован. Это приводит к дублированию логики.

**Последствия:** Два источника истины, путаница.

**Решение:** Во всех функциях handlers использовать `get_state()` и `state.set_...()`.

**Файлы:** `handlers.py` (все функции)

**Статус:** Not started

### A-02: Смешение слоёв (UI/Service/Repository)
**Описание:** Код не разделён на слои. Например, `handlers.py` одновременно обрабатывает команды, работает с БД `memory.py`, вызывает `knowledge.py` и формирует вывод через `console.print`. Это нарушает принцип разделения ответственности.

**Последствия:** Сложность тестирования, изменение одного слоя затрагивает другие.

**Решение:** Выделить:
- `ui/` — только вывод (Rich)
- `services/` — бизнес-логика (анализ, обработка)
- `repositories/` или `data/` — работа с БД, файлами
- `handlers/` — только маппинг команд → сервисы

**Файлы:** Вся кодовая база

**Статус:** Not started

### A-03: Нет зависимости от интерфейсов
**Описание:** Код жёстко привязан к конкретным реализациям (SQLite, Chroma, Ollama). Нет абстракций (репозиториев, портов).

**Последствия:** Сложно заменить компоненты (например, перейти с Chroma на FAISS).

**Решение:** Ввести интерфейсы через `abc` модуль, использовать Dependency Injection.

**Приоритет:** Low

**Статус:** Not started

## Security Issues (Безопасность)

### S-01: Command injection в practice.py
**Описание:** См. C-05.

**Приоритет:** High

**Статус:** Not started

### S-02: Нет ограничения длины ввода
**Описание:** Пользователь может ввести огромный текст (например, 100к символов), который уйдёт в LLM, может превысить лимит токенов или привести к DoS.

**Решение:** Ограничить `user_input` до 2000 символов в основном цикле (`main.py`).

**Файлы:** `main.py`

**Статус:** Not started

### S-03: Логирование potentially sensitive data
**Описание:** В `terminal_log.py` и `memory.py` логируются все команды пользователя. Если пользователь случайно введёт пароль, он сохранится в открытом виде в `memory/terminal_log.txt` и `chat_history.db`.

**Решение:** 
- Фильтровать пароли (регулярное выражение для `password=...`, `api_key=...` и т.д.)
- Или не логировать строки, содержащие определённые ключевые слова.
- Или шифровать логи.

**Файлы:** `terminal_log.py`, `memory.py`

**Статус:** Not started

### S-04: Нет rate limiting
**Описание:** Можно делать бесконечные запросы к LLM, что приведёт к перерасходу средств (если OpenRouter платный) или блокировке.

**Решение:** Простой rate limiter (tokens bucket) в `main.py` перед вызовом LLM.

**Файлы:** `main.py`, `ollama_client.py`

**Статус:** Not started

## Code Quality (Качество кода)

**Описание:** Нет статической проверки типов. Есть type hints в новых функциях, но не везде.

**Решение:** Добавить `mypy` в CI, запускать локально. Добавить аннотации во все функции.

**Файлы:** Все .py

**Статус:** Not started

### Q-02: Нет ruff/flake8
**Описание:** Нет линтера для catching неиспользуемых импортов, неопределённых переменных, сложных функций.

**Решение:** Добавить `ruff` (быстрее flake8), настроить .ruff.toml.

**Статус:** Not started

### Q-03: Смешение языков (EN/RU)
**Описание:** Комментарии, строки, вывод на русском и английском. Это затрудняет международное участие.

**Решение:** Привести всё к единому стандарту (предпочтительно английский для кода, русский для пользовательского вывода). Документировать правило.

**Файлы:** Все

**Статус:** Not started

### Q-04: Дублирование кода
**Описание:** В `handlers.py` дублируются импорты, есть неиспользуемые переменные. В `main.py` и `handlers.py` пересекается логика обработки режимов.

**Решение:** Рефакторинг: вынести общие utility functions, очистить неиспользуемый код.

**Файлы:** `handlers.py`, `main.py`

**Статус:** Partially done (улучшено при исправлении импортов)

### Q-05: Длинные функции
**Описание:** Некоторые функции в `handlers.py` и `main.py` >50 строк. Сложно читать.

**Решение:** Разбить на mehrere подфункции.

**Файлы:** `handlers.py`, `main.py`, `news_fetcher.py`

**Статус:** Not started

## Performance (Производительность)

### P-01: Полная переиндексация RAG при каждом изменении
**Описание:** В `knowledge.py` при изменении хотя бы одного PDF перестраивается вся база Chroma, хотя она поддерживает инкрементальное добавление.

**Последствия:** Медленное обновление при добавлении новых книг.

**Решение:** Использовать `chroma.add_documents` только для новых/изменённых файлов. Хранить хеши файлов в metadata.

**Файлы:** `knowledge.py`

**Статус:** Not started

### P-02: Нет лимита контекста LLM
**Описание:** В промптах может набираться большой контекст (история, RAG документы), что может превысить лимит модели (для Nemotron 3 Nano — 4096?].

**Решение:** Обрезать RAG документы, суммаризировать историю каждые N сообщений. Считать токены через tiktoken.

**Файлы:** `main.py` (промпт), `ollama_client.py`

**Статус:** Not started

### P-03: Нет асинхронности
**Описание:** Все вызовы синхронные. LLM, RAG, Docker — всё блокирует event loop.

**Решение:** Перевести на asyncio (особенно для LLM и некоторых операций с файлами).

**Статус:** Not started

## UX (Пользовательский опыт)

### U-01: Нет команды /clear уже (но есть?)
**Описание:** В чеклисте отмечено, что /clear уже есть, но в текущем коде её нет? Проверить.

**Решение:** Убедиться, что /clear присутствует и работает. Если нет — добавить.

**Файлы:** `handlers.py`

**Статус:** Unknown

### U-02: Нет /help detail
**Описание:** Команда /help показывает только список команд. Нет подробного описания каждой с примерами.

**Решение:** Добавить команду `/help <cmd>` или `/help detail` с расширенными описаниями.

**Файлы:** `ui.py`, `handlers.py`

**Статус:** Not started

### U-03: Нет sumarization истории
**Описание:** При длинном диалоге контекст растёт, LLM может терять фокус.

**Решение:** Каждые 20 сообщений генерировать краткое резюме через LLM и использовать его вместо полной истории.

**Файлы:** `memory.py`, `main.py`

**Статус:** Not started

### U-04: Кракозябры в консоли Windows
**Описание:** В Windows консоль может не поддерживать UTF-8, тогда символы (русские, эмодзи) отображаются как квадраты.

**Решение:** 
- Уже есть `PYTHONIOENCODING=utf-8` в config.py.
- Рекомендовать пользователю использовать шрифт с UTF-8 (Cascadia Code, Consolas с поддержкой).
- Или использовать colorama для корректного вывода на Windows.

**Файлы:** `config.py`, `ui.py`

**Статус:** Partially done (кодировка установлена)

### U-05: Нет graceful degradation для LLM
**Описание:** Если OpenRouter недоступен, программа падает с исключением.

**Решение:** В `LazyLoader.get_llm()` перехватывать исключения, возвращать None или заглушку, выводить сообщение "LLM временно недоступна".

**Файлы:** `config.py: LazyLoader.get_llm()`

**Статус:** Not started

### U-06: Длинные ответы LLM
**Описание:** LLM может генерировать очень длинные ответы (за Limits не следим).

**Решение:** Добавить max_tokens в параметры LLM (например, 2000).

**Файлы:** `config.py` (LLM создание), `ollama_client.py`

**Статус:** Not started

## Dependencies (Зависимости)

### Dep-01: Неполный requirements.txt
**Описание:** В `requirements.txt` указаны только некоторые зависимости. Многие модули (docker, bs4, lxml, sentence-transformers, chromadb) не зафиксированы.

**Последствия:** Нельзя воспроизвести окружение через `pip install -r requirements.txt`.

**Решение:** Сгенерировать полный список через `pip freeze > requirements.txt` или вручную добавить все импорты из кода.

**Файлы:** `requirements.txt`

**Статус:** In progress (сейчас в файле много зависимостей, но возможно не все)

### Dep-02: Нет pyproject.toml / poetry
**Описание:** Используется старый подход с requirements.txt. Лучше перейти на pyproject.toml с poetry или pip-tools.

**Решение:** Создать `pyproject.toml` с разделением core/dev зависимостей.

**Приоритет:** Low

**Статус:** Not started
